# Contributing

> Great software begins with great engineering, not great code.

---

# Welcome

Thank you for your interest in contributing to the **DeTLeng BigQuery MCP Server**.

This project is more than an MCP implementation.

It is part of the broader DeTLeng vision of combining **Data Engineering**, **Business Intelligence**, and **Artificial Intelligence** into reusable Intelligent Business Systems.

Whether you are fixing a bug, improving documentation, adding Business Intelligence tools, or proposing new ideas, your contribution is appreciated.

---

# Our Philosophy

At DeTLeng, we believe:

> **Business problems should drive technology decisions.**

Every contribution should improve one or more of the following:

- Data Engineering
- Business Intelligence
- Artificial Intelligence
- Automation
- Business Value

Technology is the tool.

Business value is the objective.

---

# Before You Write Code

Please understand the business problem first.

Ask yourself:

- What business problem does this solve?
- Will this improve decision-making?
- Can this be reused?
- Does it simplify the platform?
- Does it align with the DeTLeng architecture?

If the answer is "Yes", then you're moving in the right direction.

# Coding Standards

Every contribution should follow these principles.

✓ Readable

✓ Modular

✓ Reusable

✓ Well Documented

✓ Business Focused

✓ Production Ready

Write code for the next engineer, not just the computer.

---

# Naming Standards

Use business terminology whenever possible.

Preferred

```
get_total_revenue()

get_customer_count()

get_sales_summary()
```

Avoid

```
query1()

execute()

test()

data1()
```

The code should describe the business capability.

---

# Documentation Standards

Every new feature should include documentation.

Recommended updates:

- README
- Architecture
- Business Tools
- API Reference
- Examples

Documentation should evolve together with the implementation.

---

# Pull Requests

Before submitting a Pull Request, please verify:

- The feature works correctly.
- Documentation has been updated.
- Existing functionality is unaffected.
- Code follows project standards.
- Business value is clearly explained.

Small, focused Pull Requests are preferred over large unrelated changes.
